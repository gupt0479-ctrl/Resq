# UI and Demo Playbook

The UI should feel like one rescue workspace, not a collection of features.

## Page-by-page intent

### Landing page

Must answer in one glance:

- what pain exists
- why this is fintech
- why this is not just another dashboard

### Dashboard

Must show:

- cash stress
- receivables pressure
- open rescue actions
- latest survival-scan state

Do not overload it with secondary product stories.

### Rescue queue

This is the main working surface.

Must show:

- prioritized risks
- next action
- current state
- why each item matters now

### Financing results

Must read as real decision support, not vague research.

Show:

- lender
- product
- terms
- speed
- source
- confidence or warnings
- mode badge

### Workflow page

Must prove multi-step work happened.

Show:

- run start
- tools or checks used
- findings captured
- warnings
- final recommendation

### Integrations page

Must explain trust.

Show:

- TinyFish status
- mock/live/degraded truth
- connector health
- why the system is safe even if live dependencies fail

## Demo order

### 90-second version

1. business is under cash pressure
2. open rescue queue
3. run survival scan
4. financing options appear
5. audit timeline proves the work

### 5-minute version

1. landing or dashboard
2. cash stress and overdue receivables
3. rescue queue
4. financing options
5. vendor or insurance supporting proof
6. workflow timeline
7. integrations truth

## Design rule

Remove anything that does not strengthen the rescue story.

If a surface feels generic, restaurant-first, or unrelated to survival actions,
de-emphasize it or hide it from the demo path.
